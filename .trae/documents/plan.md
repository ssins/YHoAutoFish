# 异环自动钓鱼程序 (Auto-Fishing Bot) 完整实现方案

## 1. 需求与现状分析 (Current State & Requirements)

- **目标**: 实现基于图像识别和模拟按键的《异环》全自动钓鱼脚本。

- **环境**: Windows 系统，Python 项目（目前仅为空白模板）。

- **用户核心需求**:
  - **分辨率适配**: 采用获取游戏进程句柄 (HWND) 的方式截取游戏画面，解决分辨率和窗口位置变化问题。

  - **高响应溜鱼控制**: 结合连续点按与长按，在黄色游标处于绿色耐力条内时左右点按维持，在绿色条大范围移动时时长按跟随。要求极低迟滞。

  - **失败恢复机制**: 识别“鱼儿溜走了”提示，等待约1秒后自动恢复到初始抛竿状态。

  - **纯视觉方案**: 不使用音频辅助，完全依赖图像特征和文字特征识别。

  - **现代化GUI**: 程序需要配备现代化美观的图形用户界面。

## 2. 技术栈与依赖选择 (Tech Stack & Dependencies)

- **GUI 框架**: `customtkinter` (基于 Tkinter 封装，提供现代化的深色/浅色主题和圆角控件，极其适合写工具脚本界面)。

- **屏幕捕获**: `mss`。通过句柄获取窗口坐标后，用 `mss` 截取指定区域 (ROI)，速度极快（可达 60+ FPS），足以满足无迟滞的控制需求。

- **图像处理与识别**: `opencv-python` (cv2) + `numpy`。
  - 文字/提示框识别：使用模板匹配 (`cv2.matchTemplate`)。

  - 耐力条识别：使用 HSV 颜色过滤 (`cv2.inRange`) 和轮廓检测 (`cv2.findContours`)。

- **键鼠模拟**: `pydirectinput` 或基于 `ctypes.windll.user32.SendInput` 的底层封装。由于是 3D 游戏，必须使用底层 DirectInput 级别的模拟按键。

- **窗口控制**: `pywin32` (`win32gui`, `win32process`, `win32api`) 用于根据进程名获取窗口句柄、置顶窗口、获取客户区坐标。

## 3. 核心架构与模块划分 (Architecture & Modules)

项目建议拆分为以下几个核心模块：

- `main.py`: 程序入口，负责启动 GUI。

- `gui/`: 界面模块，包含主窗口、设置面板、运行状态显示。

- `core/`: 核心逻辑模块。
  - `window_manager.py`: 负责查找游戏进程、获取窗口句柄、计算实际客户区（Client Area）坐标，解决分辨率适配问题。

  - `screen_capture.py`: 封装 `mss`，根据 `window_manager` 提供的坐标，高频截取全屏或局部 ROI（感兴趣区域）。

  - `vision.py`: 视觉识别算法，包含找图（钓鱼按钮、上钩提示、溜走提示）和进度条解析（提取黄条和绿条的 X 坐标）。

  - `controller.py`: 键鼠操作封装，提供高精度的按下、抬起、点击方法。

  - `state_machine.py`: 自动钓鱼的主循环状态机，运行在独立线程中，避免阻塞 GUI。

## 4. 状态机设计与业务逻辑 (State Machine Logic)

定义以下几个核心状态：

- **STATE_IDLE (准备/待机)**
  - **行为**: 检测画面右下角是否存在 `F` 抛竿图标。

  - **动作**: 若存在，按 `F` 抛竿，进入 `STATE_WAITING`。

- **STATE_WAITING (等待上钩)**
  - **行为**: 循环检测屏幕中央是否出现“鱼上钩了，快点击按钮上鱼！”或检测右下角 `F` 键的高亮状态。

  - **动作**: 检测到后，立即按 `F` 键，进入 `STATE_FISHING`。

- **STATE_FISHING (溜鱼控制 - 核心难点)**
  - **行为**: 以极高频率（如 50 FPS）截取屏幕顶部偏右的“耐力条 ROI”。提取绿色目标条的中心点 `Target_X` 和黄色游标的中心点 `Cursor_X`。

  - **控制算法 (点按与长按混用)**:
    - 计算偏差 `Error = Target_X - Cursor_X`。

    - 定义死区阈值 `T_deadzone` (例如绿条宽度的一半减去少许余量)。

    - 定义长按阈值 `T_hold` (例如绿条宽度的 80%)。

    - **如果** **`abs(Error) > T_hold`**: 说明绿条跑远了，需要长按。若 `Error > 0`，长按 `D` (并释放 `A`)；若 `Error < 0`，长按 `A` (并释放 `D`)。

    - **如果** **`T_deadzone < abs(Error) <= T_hold`**: 游标在绿条边缘，需要连续点按修正。向对应方向发送短促的按键 (KeyDown + 极短Sleep + KeyUp)。

    - **如果** **`abs(Error) <= T_deadzone`**: 游标已在绿条安全区内。交替微频点按 `A` 和 `D`，或者不做任何操作（具体取决于游戏内黄条不操作时是否会自动滑落）。

  - **跳出条件**:
    1. 识别到结算界面特征 -> 进入 `STATE_RESULT`。
    2. 识别到“鱼儿溜走了”特征 -> 进入 `STATE_FAILED`。
    3. **超时保护 (Timeout)**: 如果在该状态停留超过预设最大溜鱼时间（如 3 分钟），强制进入 `STATE_FAILED` 防卡死。

- **STATE_RESULT (结算阶段)**
  - **行为**: 检测到结算界面，出现鱼的图标和重量。

  - **动作**: 模拟点击屏幕空白处关闭结算，等待 1-2 秒，回到 `STATE_IDLE`。

- **STATE_FAILED (失败处理)**
  - **行为**: 检测到“鱼儿溜走了”或触发超时。

  - **动作**: 停止一切按键（释放 A/D），等待约 1.5 秒，确认画面回到初始状态后，进入 `STATE_IDLE` 重新抛竿。

## 5. 关键技术解决方案 (Key Technical Solutions)

### 5.1 进程分辨率适配方案

- 使用 `win32process` 枚举进程找到 `HTGame.exe`，进而获取《异环》的游戏窗口句柄 (HWND)。这种方式比单纯匹配窗口标题更稳定，防伪装。
- 使用 `win32gui.GetClientRect` 和 `win32gui.ClientToScreen` 获取**纯游戏画面**在屏幕上的绝对坐标和真实分辨率 (Width, Height)。

- **坐标相对化**: 所有的截图区域 (ROI) 坐标不写死绝对值，而是写成相对于 (Width, Height) 的比例或直接在获取到真实分辨率后自动缩放计算。例如，耐力条中心在宽度的 50%-70%，高度的 10%-15% 区域。这样无论窗口多大，都能精准截取。

### 5.2 无迟滞的溜鱼控制方案与算法优化

- **独立线程**: 视觉识别和控制循环必须在独立的后台 `Thread` 运行，内部使用极小的 `time.sleep(0.01)` 甚至 `time.sleep(0)` 配合高精度计时器，确保控制帧率在 60 FPS 左右。

- **状态记录**: 记录当前 `A` 和 `D` 键的物理按下状态。当需要从“长按”切换到“点按”时，先强制发送 `KeyUp`，避免按键冲突。

- **惯性与防震荡 (Momentum & Anti-Jitter)**: 考虑到游戏内游标移动的物理惯性，引入简单的阻尼或微分 (Derivative) 计算。除了计算偏差 `Error` 外，计算其变化率 `dError`，当发现游标以较快速度接近目标中心时，即使还在死区外，也应提前释放按键或转为反向微点按，以防过冲 (Overshoot)。

- **焦点保护机制**: 在发送每一次按键前，通过 `win32gui.GetForegroundWindow()` 检查当前前台窗口是否为《异环》。若用户切出游戏，立即停止所有模拟按键并进入暂停状态，防止干扰其他操作。

### 5.3 视觉识别鲁棒性提升

- **HSV 颜色空间**: 耐力条的绿色和黄色在 RGB 下容易受背景干扰，转换到 HSV 色彩空间后，提取特定范围的色调 (Hue) 极其稳定，计算速度在 1 毫秒内，杜绝识别延迟。

- **形态学过滤 (Morphological Filtering)**: 在生成颜色掩码 (Mask) 后，为防止背景中恰好有相似颜色的光影造成误判，对轮廓 (Contours) 进行长宽比、最小面积和最大面积的严格过滤。只提取符合“竖条”和“横条”几何特征的像素块作为有效坐标。

### 5.4 现代化 GUI 深度优化方案 (Advanced GUI & UX Design)

为确保脚本的易用性、可调试性及长期运行的稳定性，GUI（基于 `customtkinter`）需进行如下深度优化设计，以解决“参数难调、容易假死、内存泄露”等挂机脚本通病：

- **5.4.1 核心视图划分 (View Layout)**
  - **主控看板 (Dashboard)**:
    - 包含醒目的状态指示灯（绿色: 运行中 / 红色: 已停止 / 黄色: 异常中断）。
    - 全局统计卡片：显示已钓鱼数量、已运行时长、平均每条耗时。
    - “启动 (F9)” 和 “停止 (F10)” 主控按钮。

  - **参数配置 (Settings)**: 暴露核心调节参数，解决不同设备环境的差异：
    - **控制参数**: 长按阈值 (`T_hold`)、死区大小 (`T_deadzone`)、点按频率。
    - **热键设置**: 允许用户自定义启动、停止热键，避免与其他软件冲突。
    - **通用开关**: “窗口置顶” (Always on top) 开关、“保存日志到文件”开关。

  - **视觉标定与调试 (Calibration & Debug)**:
    - **参数微调滑块**: 提供 H、S、V (色调/饱和度/明度) 的 `min` / `max` 调节滑块，以应对游戏昼夜交替或不同水域的光影变化导致进度条颜色偏差。
    - **视觉预览开关 (Debug View)**: 开启后弹出一个小窗，实时显示 OpenCV 截取到的画面及识别框（绿条/黄条被框出），让用户能“看懂机器人的眼睛”，极大降低排错难度。

- **5.4.2 潜在问题规避与稳定性设计 (Robustness in GUI)**
  - **UI 线程防假死 (Thread-Safety)**: 视觉识别与按键控制严格在后台 `Thread` 运行。后台通过线程安全的 `queue.Queue` 或系统事件向主线程发送状态更新（如日志文本），GUI 仅负责定时读取队列并渲染，彻底杜绝点击“启动”后窗口拖不动、无法响应的假死问题。
  - **日志内存防溢出 (Memory Leak Protection)**: 实时日志框必须设置最大行数限制（如通过双向队列 `deque` 限制最多保留 500 行）。若不限制，挂机一整晚后文本框会占用数 GB 内存导致程序崩溃。
  - **配置持久化 (State Persistence)**: 用户修改的所有 UI 设置（热键、HSV 阈值、置顶状态等）在程序关闭时自动序列化保存至 `config.json`，下次启动时自动回显，实现“开箱即用”。
  - **防误触机制 (Safe Mode)**: 在脚本启动进入工作状态后，必须自动“禁用 (Disable)”敏感设置项（如修改热键、调节 HSV 参数），必须先点击“停止”才可修改，防止在运行中更改参数导致逻辑崩溃。

## 6. 验证与测试步骤 (Verification Steps)

1. **窗口绑定测试**: 启动程序，在日志中确认是否正确获取到了《异环》的窗口句柄和真实的客户区分辨率。
2. **ROI 截取测试 (Debug 模式)**: 开启 OpenCV 的预览窗口，确认截取的“右上角耐力条区域”、“中心文字区域”是否准确。
3. **颜色过滤测试**: 在溜鱼时，验证 HSV 掩码 (Mask) 是否能干净地提取出绿色目标框和黄色游标，并正确画出中心点标记。
4. **控制环路测试**: 模拟极端的鱼漂移动，观察程序输出的按键指令日志，是否符合“大偏差长按、小偏差点按”的逻辑。
